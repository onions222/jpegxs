# LaTeX 初学者写作风格指南（ElegantBook 版）

本文件用于约束 `docs/jpegxs-textbook-latex/` 的教学写法。默认假设教材使用 **ElegantBook** 文档类。目标读者是 JPEG XS 初学者，因此重点不是“把所有信息都塞进去”，而是：

1. 先建立直觉
2. 再定义变量和术语
3. 再进入公式
4. 最后再映射到实现

## 总原则

- 初学者第一次看到一个概念时，必须先知道“它是什么 / 为什么需要它”
- 变量必须先定义再使用
- 公式必须先讲输入输出对象，再给表达式
- 实现映射是帮助理解，不是代替理论说明
- 不要默认读者已经懂图像压缩、小波、bitplane、码率控制
- 要利用 ElegantBook 的“书籍式阅读节奏”，让章节更像教材而不是技术备忘录

## 推荐讲解顺序

对于复杂章节，优先采用以下顺序：

1. 一句话直觉
2. 目标问题
3. 最小例子
4. 正式定义
5. 数学表达
6. 实现映射
7. 边界条件 / 常见误解

### 示例：GCLI

不要一开始就写 predictor、residual、sigflag。  
应先写：

- GCLI 描述的是“一组系数最高用到了哪一层 bitplane”
- 它不是在编码精确系数，而是在给后续编码提供位深摘要

然后再给最小例子和正式定义。

## 变量定义要求

### 必须做到

- 变量首次出现时，附近必须能找到定义
- 同一章内不要随意更换同一对象的记号
- 若变量来自标准或代码，必须说明它是“数学符号”还是“实现变量名”

### 推荐写法

```tex
设当前子带系数为 $c_i$，其中 $i$ 表示组内索引。令
\[
M = \bigvee_i |c_i|,
\]
则 $M$ 表示该组系数幅值按位 OR 之后得到的聚合结果。
```

### 避免写法

- 直接给出 `$M = ...$`，但不说 $M$ 代表什么
- 一会儿写 `gcli`，一会儿写 `GCLI`，一会儿又写 “位深”

## 参数解释要求

对于配置类章节，不能只给表格。要先用自然语言解释每类参数在控制什么。

### Profile

必须先解释：

- Profile 限定的是“允许使用哪些编码工具和信号路径”
- 它不是分辨率上限，也不是码率上限

### Level

必须先解释：

- Level 限定的是尺寸、样本数、吞吐量等上限
- `W_{\max}`、`H_{\max}`、`L_{\max}` 分别限制什么

### Sublevel

必须先解释：

- Sublevel 主要控制码率上限
- 它与 Profile / Level 是互补关系

## 公式表达要求

### 公式前

先回答：

1. 输入是什么
2. 输出是什么
3. 为什么要这样定义

### 公式后

至少跟一句自然语言解释，例如：

- 这个式子说明……
- 该表达式的作用是……
- 这里的右移相当于整数域中的……

### 整数实现相关

涉及整数近似时，必须明确：

- 向下取整
- 四舍五入
- 饱和裁剪
- 位移对应的尺度变化

## 最小例子要求

每个难章节都建议在正文中保留一个最小例子。

### 适合保留最小例子的章节

- NLT
- MCT
- DWT
- GCLI
- 码率控制
- Packing

### 最小例子的作用

- 帮读者先抓住对象
- 让后面的正式公式有“落点”
- 降低一次性引入多个变量的压力

## 工程解释与代码事实

### 代码直接事实

适用于：

- 函数调用顺序
- 条件分支
- 位移和循环
- 数据布局

这类内容正常叙述，并在 `codefact` / `sourcetrace` 中标来源。

### 工程解释 / 实现推断

适用于：

- 为什么作者可能这样设计
- 为什么这种布局更适合硬件或缓存
- 为什么先做 A 再做 B 更自然

这类内容必须与代码直接事实区分，可使用：

- `\begin{engineeringnote}...\end{engineeringnote}`
- 或句首标 `工程解释：`
- 或句首标 `实现推断：`

## 初学者常见误区

写作时应主动防守这些误区：

1. 把 GCLI 和 GTLI 混为一谈
2. 把 Profile 当成分辨率限制
3. 把 Level 当成单一参数
4. 把 precinct 和 packet 混为一谈
5. 把“数学定义”和“实现变量名”混为一谈

建议在每章的“常见误解”节里至少覆盖 1--3 个高频误区。

## 章节内部的阅读节奏

一节内容尽量遵循：

1. 先用 1 段讲直觉
2. 再用表或变量表定义对象
3. 再给 1--3 个关键公式
4. 再讲实现映射
5. 最后给一个小结

不要出现以下节奏：

- 上来一屏公式
- 上来一屏函数名
- 上来一个超大表格，正文不解释怎么读

## 何时把内容放回正文

如果出现以下情况，应把解释从图表挪回正文：

- 图节点文字超过两行且仍然讲不清
- 表格说明列太长
- 一个公式需要配三段解释
- 算例中的变量定义比计算本身还长

## 自检问题

写完一节后，至少问自己：

1. 初学者第一次读这节，能否用一句话说出“这节在解决什么问题”
2. 每个公式附近是否都有变量定义
3. 读者是否会把两个容易混淆的概念搞混
4. 是否先给了直觉，再给了正式定义
5. 是否把实现细节过早地压到了读者脸上
