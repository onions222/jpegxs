# 教材生命周期工作流（ElegantBook / JPEG XS）

本文件定义这套教材从“创建”到“修复”再到“可用”的完整流程。目标不是一次性生成长文，而是通过多轮收敛，把教材从初稿推进到可学习、可核验、可维护的状态。

## 总原则

- **源码是唯一真值来源**：标准语义可以解释，但实现细节必须回到本地代码核对
- **先搭结构，再补细节**：不要一开始就在局部公式上陷太久
- **先让主线完整，再追求优雅**
- **先正确，再通俗**
- **先主文一致，再算例一致，再端到端一致**

## Phase 0 — 约束冻结

在动笔前先固定这些内容：

1. 目标读者：默认是 JPEG XS 初学者
2. 真值来源：
   - `libjxs/src/`
   - `libjxs/public/`
3. 目标工程：
   - `docs/jpegxs-textbook-latex/`
4. 默认模板：
   - **ElegantBook**
5. 统一风格：
   - 中文正文
   - 英文术语/函数名保留
   - 变量先定义再使用
6. 参考规则：
   - `chapter-template.md`
   - `latex-table-and-figure-style.md`
   - `latex-beginner-writing-style.md`
   - `repo-mapping-guide.md`
   - `worked-example-rules.md`

如果这一步没有冻结，后面容易反复返工。

## Phase 1 — 工程骨架

目标：先让教材“像一本书”，而不是一堆零散 tex。

需要完成：

1. `main.tex`
2. `preamble.tex`
3. `chapters/`
4. `examples/`
5. `appendices/`
6. `README.md`
7. 最小可编译 PDF

这一阶段关注：

- include 结构
- 章节顺序
- 盒子环境
- 表格/流程图基本可编译

这一阶段不追求：

- 每个公式都完美
- 每个解释都对初学者最友好
- 所有细节都与代码逐行核验

## Phase 2 — 首版内容生成

目标：先把主链讲全。

推荐先完成这些章节：

1. 00 符号与约定
2. 01 概述
3. 03 配置
4. 04 NLT
5. 05 MCT
6. 06 DWT
7. 07 Bands / Precincts
8. 08 GCLI
9. 09 Rate Control
10. 10 Quantization
11. 11 Packing
12. 12 Decoder
13. 14 End-to-end trace

这一阶段的要求：

- 每章都有基本结构
- 关键公式先写出来
- 至少有基本实现映射
- 至少有一个正文最小例子

这一阶段允许存在：

- 初学者不够顺
- 少量概括过粗
- 交叉章节还没完全同步

## Phase 3 — 初学者友好化

目标：把“工程参考稿”变成“可学教材”。

重点动作：

1. 给难章节补一句话直觉
2. 给每章补“最小例子”
3. 给变量补首次定义
4. 给表格补“如何读这张表”
5. 把“术语轰炸”拆成层次化解释

重点章节通常包括：

- 03 Configuration
- 06 DWT
- 08 GCLI
- 09 Rate Control
- 11 Packing

常见问题信号：

- 用户说“看不懂为什么要有这个东西”
- 用户分不清 GCLI / GTLI / coefficient
- 用户分不清 Profile / Level / Sublevel
- 用户能看懂表，但不知道表在流程里起什么作用

## Phase 4 — 内容正确性修订

目标：修“会把读者带偏”的错误。

这一步不是普通润色，而是**按代码审书**。重点排查：

1. 错误概括
2. 错公式
3. 错单位
4. 错机制归因
5. 把局部实现当成全局规律
6. 把 fallback 说成主路径
7. 把组级和块级粒度混淆

高风险章节：

- 03 Configuration
- 07 Bands / Precincts
- 08 GCLI
- 09 Rate Control
- 10 Quantization
- 11 Packing
- 15 Notes

这一阶段通常要产出“修正清单”，例如：

- `raw fallback` 是否被误说成 Q/R 搜索兜底
- `sigflag` 粒度是否讲对
- 子带公式是否正确
- 配置家族是否被过度概括
- 单位是 byte / nibble / bit 哪一个

## Phase 5 — 全书一致性审校

目标：修“一处改了，别处还在说旧话”。

检查顺序建议：

1. `08 -> 09 -> 10 -> 11 -> 12`
2. `03 / 07`
3. `14 end-to-end trace`
4. 算例册
5. `15 notes`

重点检查：

1. 主文与算例是否一致
2. 主文与 trace 是否一致
3. 主文与 notes 是否一致
4. 同一公式在多个章节是否一致
5. 同一术语在不同章节是否换了意思

典型要同步的对象：

- raw fallback
- sigflag
- GCLI / GTLI
- 列宽公式
- 子带总数公式
- DWT 可逆性的前提
- 码率控制失败路径

## Phase 6 — 可用性验收

目标：判断教材是否达到“可用”。

验收标准不是“排版漂亮”，而是：

1. **说明清楚**：每章能回答“这一步在解决什么问题”
2. **说明白**：初学者能跟上主链
3. **说得对**：关键概念、公式、单位、机制与代码一致

至少要通过这几类检查：

- 主线是否能一口气读通
- 难章节是否有最小例子
- 算例是否可复算
- 术语是否前后一致
- 是否还有关键误导性说法

## 哪些问题在哪一阶段处理

### 属于 Phase 1/2 的问题

- 缺章节
- 缺 include
- 没有公式
- 没有实现映射

### 属于 Phase 3 的问题

- 看不懂
- 太抽象
- 变量没定义
- 表能看见但不会读

### 属于 Phase 4 的问题

- 讲错了
- 公式不对
- 单位不对
- 粒度不对
- fallback 路径说错

### 属于 Phase 5 的问题

- 这一章改了，另一章没改
- 主文改了，算例没改
- 端到端 trace 还在沿用旧说法

### 属于 Phase 6 的问题

- 读完还是抓不住主线
- 仍然存在关键误导
- 作为教材还不敢交给初学者

## 实战建议

### 不要过早做的事

- 不要在首版内容还没齐时就陷入全书排版抛光
- 不要在主文概念还不稳时就写太重的 trace
- 不要先把算例写得比主文还复杂

### 应该尽早做的事

- 尽早建立 08/09/10/11/12 的主链一致性
- 尽早修正会建立错误心智模型的说法
- 尽早给配置类和 GCLI 类章节补最小直觉

## 何时算“基本可用”

满足以下条件时，可以说教材“基本可用”：

1. 主链齐全
2. 关键难章有最小例子
3. 高风险错误已修
4. 主文、算例、trace 基本同步
5. 读者不会因为错误概括而建立反向心智模型

## 何时算“接近终稿”

在“基本可用”基础上，再满足：

1. notes 不再残留旧说法
2. 算例册和主文完全同步
3. 配置、GCLI、码率控制这三处没有明显概念风险
4. 端到端 trace 与修正后的主文一致

这时才适合把工作重心从“内容正确性”切换到“排版和精修”。
